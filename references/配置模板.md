# 配置模板 — 飞书多Agent跨实例协作

> **⚠️ 使用前必须填写！**
> 将下方占位符替换为你的实际值，否则协作无法工作。
> 填写后运行 `python scripts/validate_config.py` 校验配置。

---

## 一、飞书群信息

```yaml
group_id: "oc_你的群ID"        # 例：oc_a5a1e854b61386229ace21a8ba2c97fb
group_name: "你的群名"         # 例：陈飞飞
```

**如何获取群ID？**
- 在飞书中打开群设置 → 群信息 → 复制群ID

---

## 二、Bot身份配置

### Bot A（主持者）

```yaml
bot_a:
  name: "菲菲"                  # 在群里显示的名字
  open_id: "ou_你的open_id"     # 例：ou_fc4fbfecb748e0093462334de71e669a
  cli_session_id: "cli_你的cli_id"  # 例：cli_a9557440a9b8dcce（仅用于私下路由，不用@标签）
  role: "主持者"                 # 固定：主持者
  responsibilities:
    - "接收用户指令"
    - "分析任务、拆解步骤"
    - "@执行者派发具体操作"
    - "把关合规、审核结果"
    - "代表用户做最终决策"
```

### Bot B（执行者）

```yaml
bot_b:
  name: "珠珠"                  # 在群里显示的名字
  open_id: "ou_你的open_id"     # 例：ou_47eb6e4cddfcebaa8f1150a16e88713b
  cli_session_id: "cli_你的cli_id"  # 例：cli_a9637d13d4f95bb3
  role: "执行者"                # 固定：执行者
  responsibilities:
    - "接收主持者@的任务"
    - "执行信息采集/技术操作"
    - "完成后@主持者汇报结果"
    - "不直接响应用户"
```

### Bot C（可选，扩展角色）

```yaml
bot_c:
  name: "你的第三个Bot名字"
  open_id: "ou_你的open_id"
  cli_session_id: "cli_你的cli_id"
  role: "执行者"                # 可选：执行者 / 协作员 / 审计员
  responsibilities:
    - "按需扩展"
```

## 三、Bot之间互相@用的ID对照表

| 发送方 | @接收方用open_id |
|--------|------------------|
| Bot A（主持者）@ Bot B（执行者） | `ou_你的bot_b_open_id` |
| Bot B（执行者）@ Bot A（主持者） | `ou_你的bot_a_open_id` |

## 四、需要写入每个Bot的MEMORY中的配置

### Bot A（主持者）需要记住：

```
【飞书协作ID — Bot A视角】
@Bot B用open_id：ou_你的bot_b_open_id
Bot B的名字：你的bot_b名字
Bot A自己的open_id：ou_你的bot_a_open_id
Bot A自己的cli_会话ID：cli_你的bot_a_cli_id
群ID：oc_你的群ID
```

### Bot B（执行者）需要记住：

```

@Bot A用open_id：ou_你的bot_a_open_id
Bot A的名字：你的bot_a名字
Bot B自己的open_id：ou_你的bot_b_open_id
Bot B自己的cli_会话ID：cli_你的bot_b_cli_id
群ID：oc_你的群ID
```

## 五、协作协议示例

```
【协作协议 v1.0】

一、角色定义
- 主持者：负责接收用户指令、分析任务、拆解步骤、@执行者派发任务、把关合规、代表用户决策
- 执行者：负责接收主持者@的任务、执行信息采集/技术操作、完成后台@主持者汇报

二、通信方式
- 主通道：飞书群（群ID：oc_你的群ID）
- 通信格式：<at user_id="ou_对方的open_id">对方名字</at> 消息内容
- user_id必须为open_id（ou_开头），不能用cli_会话ID

三、协作流程
1. 用户在群里发指令
2. 主持者接收，分析后@执行者派发任务
3. 执行者完成任务，@主持者汇报结果
4. 主持者整理后回复用户（或直接让用户看到执行者的汇报）

四、注意事项
- 执行者不直接响应用户，汇报给主持者转发
- 重大事项需双方确认后上报用户
- 所有对外内容由主持者把关
```
